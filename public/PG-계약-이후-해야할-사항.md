# PG사 계약 이후 해야 할 사항

이 문서는 현재 **모의 구독(클라이언트에서 Firestore만 갱신)** 구조를 전제로, **실제 결제(PG) 연동** 시 진행할 작업을 정리한 것입니다.

---

## 1. 사전에 정할 것 (비즈니스·계약)

- **과금 방식**: 월 정액 **자동결제(구독)** 인지, 매월 **수동 결제** 인지, **빌링키(카드 등록 후 자동 청구)** 지원 여부.
- **PG 또는 결제 대행**: 국내 예시로 토스페이먼츠, KG이니시스, 나이스페이 등. 여러 채널을 한 번에 쓰려면 **포트원(구 아임포트)** 같은 **결제 대행(aggregation)** 을 검토할 수 있음.
- **환불·해지 정책**: 중도 해지 시 당월 이용분, 다음 결제일, 환불 가능 여부.
- **세금계산서·현금영수증** 필요 여부(법인/개인 사업자 등).

→ 위가 정해져야 **구독 상태(`subscriptionActive` 등)를 언제 `true`/`false`로 둘지**가 명확해집니다.

---

## 2. 구현 관점에서의 핵심 변화

현재는 **브라우저에서 Firestore `users` 문서만 갱신**해도 구독이 활성화됩니다.

PG 연동 후에는 반드시:

- **실제 결제 성공은 서버(또는 PG 웹훅)에서만 신뢰**하고,
- 그때만 `subscriptionActive` 및 **PG 식별자**를 저장해야 합니다.

즉, **「결제 완료 = 서버가 검증한 한 번의 이벤트」** 로 모델을 바꾸는 것이 핵심입니다.

---

## 3. 권장 아키텍처 (Next.js + Firebase 전제)

### 3.1 결제 시작

- 프론트: 구독 버튼 → **백엔드 API** 호출(또는 PG가 제공하는 **결제창 URL / 클라이언트 키**로 SDK 오픈).
- 서버: 주문 ID(`merchant_uid` 등) 생성, 금액·상품명·사용자 식별(Firebase `uid` 등)과 함께 PG에 넘길 파라미터 준비.

### 3.2 결제 완료

- PG는 **return URL**(브라우저 리다이렉트)과 **서버 노티피케이션(웹훅)** 을 제공하는 경우가 많음.
- **웹훅에서 최종 승인 검증**(금액·주문번호·상태) 후 Firestore 업데이트가 안전합니다.  
  리다이렉트만 믿으면 조작·누락·중복 처리 위험이 있습니다.

### 3.3 구독(자동결제)인 경우

- **빌링키** 발급 → 매월 PG가 청구하거나, **스케줄러**(예: 클라우드 스케줄러 + 서버 함수)가 빌링키로 결제 요청.
- 실패 시 재시도·연체·구독 중지 정책을 사전에 정해야 합니다.

### 3.4 Firestore 데이터 확장 예시

- `pgProvider`, `customerKey` / `billingKey`(또는 PG별 토큰)
- `subscriptionStatus`(예: `active`, `past_due`, `canceled`)
- `currentPeriodEnd`, `lastPaymentAt`, `lastPaymentId`, `cancelAtPeriodEnd` 등

**주의**: 카드 번호·빌링키 등은 **클라이언트·Firestore에 평문 저장 금지**. PG 권장 방식 또는 **서버 전용** 저장을 따릅니다.

### 3.5 기존 코드와의 연결

- 구독 여부는 이미 `subscriptionActive` 등으로 판단 가능하므로, PG 도입 후에는 이 필드를 **웹훅/서버 API에서만 갱신**하도록 옮기고, 클라이언트가 직접 `updateDoc`으로 구독을 켜는 경로는 **제거하거나 관리자 전용**으로 제한하는 것이 좋습니다.
- **관리자 API**(Firebase Admin + `ADMIN_UIDS`)는 환불·수동 구독 조정 등으로 확장하기 좋습니다.
- **AI API 구독 검증**은 기존처럼 서버에서 `users` 문서를 읽으면 됩니다(필드만 풍부해짐).

---

## 4. 배포(Vercel 등) 시 체크

- PG·웹훅 URL을 **HTTPS 고정 도메인**으로 등록.
- **시크릿 키·웹훅 시그니처**는 **환경 변수**만 사용(저장소에 커밋 금지).
- 웹훅은 **재전송·중복**이 올 수 있으므로 **멱등 처리**(동일 `paymentId` 중복 반영 방지).

---

## 5. 실무 체크리스트

- [ ] PG **샌드박스**로 결제·웹훅·실패 케이스 테스트  
- [ ] 구독 해지 시: **즉시 종료** vs **말일까지 이용** 정책을 서비스 약관·UI에 반영  
- [ ] 개인정보·결제 로그 보관 기간·**약관 동의** UI  
- [ ] 장애 시: PG 대시보드와 **Firestore 구독 상태 불일치** 복구 절차  

---

## 6. 요약

PG 계약 후에는 **「결제 성공 / 실패 / 정기 청구」를 웹훅·서버에서 검증**하고, 그 결과만 Firestore 구독 필드를 변경하도록 바꾸면 됩니다.  
현재 구현된 **구독 UI·AI 서비스 잠금·관리자** 흐름은 유지하고, **결제 부분만 실제 PG에 맞게 교체**하는 방향이 자연스럽습니다.
